Narrative

Artifact Description. The artifact is an Android Inventory Manager application I developed in CS 360: Mobile Architecture and Programming. The application was written in Java and targets the Android platform. It allows users to create an account, log in, and manage a personal inventory of items — each with a name, SKU, quantity, and category. Users can add new items, update quantities, delete records, and receive SMS notifications when stock falls below a defined threshold. The application was originally created during CS 360 and represented a complete, functional Android app, making it a strong candidate for enhancement across all three ePortfolio categories.

Justification and Improvements. I selected this artifact because it is the most direct demonstration of my target career skill set — Android mobile development in Java — and because it had clear, meaningful room for improvement in each of the three computer science categories. For the software engineering enhancement specifically, the original application had a significant architectural limitation: all business logic, database access, and UI rendering were handled directly inside the Activity classes. This violates the single-responsibility principle and makes the code difficult to test, maintain, or extend.

The enhancement I implemented transformed the application's architecture in several concrete ways. First, I introduced the MVVM pattern by creating an InventoryViewModel class that holds all business logic and exposes the item list to the UI through LiveData. The DashboardActivity now simply observes the LiveData stream and updates the screen when data changes, rather than managing database cursors directly. Second, I replaced the original LinearLayout inflation loop — which created every list item view at once regardless of whether it was on screen — with a RecyclerView backed by a custom InventoryAdapter. RecyclerView only creates as many view objects as fit on screen and reuses them as the user scrolls, making the list significantly more memory-efficient and scalable. Third, I implemented session management by adding a getUserId() method to DatabaseHelper and passing the logged-in user's ID through Intent extras between screens. This means every screen now knows which user is active, and inventory items are filtered and stored by owner rather than shared globally. Fourth, all column name constants in DatabaseHelper were made public so that other classes reference them directly rather than duplicating string literals, eliminating a class of potential runtime errors. Fifth, I added an owner_id foreign key column to the inventory table and wrote a safe onUpgrade migration that preserves existing data rather than dropping and recreating tables. Sixth, I consolidated the redundant NotificationActivity — its SMS logic was duplicated from DashboardActivity without adding any value — and removed it entirely. Finally, I added Javadoc documentation to every public class and method, and wrote fourteen instrumented unit tests covering all of DatabaseHelper's public methods, including edge cases like duplicate usernames, negative quantities, and non-existent record deletions.

Course Outcomes. In Module One I planned for this enhancement to address Course Outcome 4 — demonstrating the ability to use well-founded and innovative techniques, skills, and tools in computing practices to implement solutions that deliver value and accomplish industry-specific goals — and Course Outcome 2 — designing and delivering professional-quality written communications that are coherent and technically sound.

I believe both outcomes were met. The MVVM refactor, RecyclerView adoption, session management implementation, and safe database migration all reflect established, industry-standard Android development practices. These are not superficial changes; they represent the architectural decisions a professional Android developer would make when taking a proof-of-concept application toward production quality. The Javadoc documentation and unit tests directly support Outcome 2 by making the codebase readable and verifiable for any developer who encounters it.

I do have one update to my outcome-coverage plan. I originally noted that Course Outcome 1 — building collaborative environments that enable diverse audiences to support organizational decision making — would be addressed primarily through the code review and self-assessment. Having now added thorough Javadoc to every class and method, I believe this enhancement also partially contributes to Outcome 1, since well-documented code is itself a collaborative artifact that enables other developers to understand, maintain, and build on the work.

Reflection. The most significant learning experience in this enhancement was internalizing the practical difference between MVVM as a concept and MVVM as an implementation. I understood the pattern in theory before starting, but working through the actual wiring — creating the ViewModel, connecting it to LiveData, and having the Activity observe that data rather than fetch it directly — required a much more careful understanding of Android's lifecycle than I had going in. The moment that required the most thought was realizing that loadItemsForUser() needed to be called in onResume() as well as onCreate(), because when a user adds an item in AddItemActivity and navigates back, the Dashboard Activity is not recreated — it resumes. Without the onResume call, the new item would be saved to the database but never appear in the list, which was exactly the bug I encountered during testing.

The RecyclerView adapter was also more involved than I anticipated. The original LinearLayout approach felt simpler on the surface because everything happened in one place, but that simplicity was deceptive — it was hiding a scalability problem. Writing a proper ViewHolder and binding pattern forced me to think more carefully about the separation between data and presentation, which reinforced why the MVVM pattern exists in the first place.

The biggest challenge overall was the database migration. Adding the owner_id column to an existing schema required writing an onUpgrade method that altered the table rather than dropping and recreating it. Getting the SQL syntax correct for SQLite's ALTER TABLE ADD COLUMN and ensuring the default value was set appropriately took careful attention. This was a good reminder that in real-world development, schema changes must be handled with far more care than in initial development, because existing user data cannot simply be discarded.

By the end of this enhancement, the application had evolved from a functional but architecturally flat student project into a structured, maintainable codebase that reflects how professional Android applications are actually built. That gap between “it works” and “it's well-engineered” is something I now understand much more concretely than I did before.
