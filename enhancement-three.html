Outcome 1

Narrative

Artifact Description. The artifact is the Android Inventory Manager application originally developed in CS 360: Mobile Architecture and Programming. Written in Java for the Android platform, the application allows users to create an account, log in, and manage a personal inventory of items with fields for name, SKU, quantity, and category. The application was originally created during CS 360 and has been progressively enhanced across three categories for the CS 499 capstone ePortfolio, with this milestone representing the final planned enhancement focused on database design and security.

Justification and Improvements. I selected this artifact for the databases category because the original application's data layer had significant and well-defined weaknesses that provided genuine opportunities to demonstrate database design knowledge rather than superficial changes. The original schema was flat and unnormalized, passwords were stored in plaintext, there was no access control beyond a basic username check, and while the queries used parameterized placeholders in some places, the implementation had not been systematically audited for consistency. Each of these issues maps directly to a core database competency, making this artifact an ideal vehicle for demonstrating growth in that area.

The enhancement introduced four concrete improvements. First, the schema was normalized to Third Normal Form by creating a separate categories table and a suppliers table, with the inventory table referencing both via foreign keys. Previously, category was stored as a free-text string column directly in the inventory table, meaning the same category could be stored as “Electronics,” “electronics,” or “ELECTRONICS” as three separate unrelated values with no enforcement of consistency. Under the new schema, categories exist as unique named records in their own table, and inventory items reference them by foreign key ID. A helper method, getOrCreateCategory(), resolves a category name to its ID idempotently — creating the category if it does not exist, or returning the existing ID if it does — so the UI experience is unchanged while the underlying data integrity is enforced. Second, passwords are now hashed using bcrypt before being stored. The original application stored the exact string the user typed as their password, meaning anyone with access to the database file — possible on a rooted Android device — could read every user's credentials immediately. The enhanced implementation calls BCrypt.hashpw() with a work factor of 12 at registration time, and BCrypt.checkpw() at login time to compare the plaintext input against the stored hash without ever reversing it. Bcrypt was specifically chosen because it is intentionally slow and incorporates a per-password random salt, which makes both brute-force attacks and rainbow table attacks computationally infeasible. Third, role-based access control was implemented by adding a role column to the users table with two possible values: “user” and “admin.” The ViewModel now calls getUserRole() after loading data and routes the query accordingly — admin users receive all inventory items via getAllItems(), while standard users receive only their own items via getItemsByUser(). This means access control is enforced at the data access layer rather than only at the UI layer, which is a meaningful architectural distinction. Fourth, all database queries were audited and confirmed to use parameterized placeholders, with every user-supplied value passed as a bound argument array rather than concatenated into the SQL string directly. Foreign key enforcement was also enabled explicitly via PRAGMA foreign_keys = ON in both onOpen() and onCreate(), since SQLite does not enforce foreign keys by default. The onUpgrade() method was carefully written to perform all schema changes through ALTER TABLE and INSERT OR IGNORE INTO statements rather than dropping and recreating tables, preserving all existing user data across version increments.

Course Outcomes. In Module One I planned for this enhancement to address Course Outcome 4 — demonstrating the ability to use well-founded and innovative techniques, skills, and tools in computing practices to implement solutions that deliver value and accomplish industry-specific goals — and Course Outcome 5 — developing a security mindset that anticipates adversarial exploits in software architecture and designs to expose potential vulnerabilities, mitigate design flaws, and ensure privacy and enhanced security of data and resources.

Both outcomes were met. Course Outcome 4 is addressed by applying established relational database design principles — normalization, foreign key constraints, and referential integrity — to create a more maintainable and scalable data layer. These are not experimental approaches; they are the standard practices used in professional software development, and applying them to a mobile application demonstrates the ability to bring industry-standard database knowledge to bear on a real project. Course Outcome 5 is addressed most directly by the bcrypt implementation and the parameterized query audit. The plaintext password storage issue was identified in the code review as the most critical security vulnerability in the application, and resolving it required not just adding a library call but understanding why bcrypt is the appropriate choice — its work factor, its salting behavior, and the specific attacks it defends against. The role-based access control implementation addresses a different dimension of security: authorization rather than authentication, ensuring that even a logged-in user cannot access data they do not own.

With this enhancement complete, all five course outcomes are now addressed across the three milestones. Outcomes 3 and 4 are covered by the algorithms and software engineering enhancements respectively, Outcome 5 is covered here, and Outcomes 1 and 2 are addressed through the code review, narratives, and documentation produced throughout the course.

Reflection. The most significant learning from this enhancement was developing a concrete understanding of the difference between authentication and authorization, and how they must be implemented at different layers of an application. Before this project, I understood these terms conceptually, but implementing bcrypt for authentication and RBAC for authorization in the same codebase made the distinction tangible. Authentication — verifying that a user is who they claim to be — is handled at the point of login by comparing the plaintext input against the stored hash. Authorization — deciding what a verified user is allowed to do — is handled at the data access layer by routing queries based on role. Conflating these two concerns, or handling either of them only at the UI layer, would leave meaningful security gaps.

The bcrypt implementation itself surfaced an interesting challenge. My initial assumption was that password hashing would be straightforward to add, but I quickly discovered that bcrypt's intentional computational cost — the property that makes it secure — also means that login verification takes noticeably longer than a simple string comparison. At a work factor of 12, each verification call takes roughly 300 to 400 milliseconds on a modern device. For a desktop application this would be invisible, but on a mobile device running on the main thread it is perceptible. The correct solution is to move the bcrypt verification call to a background thread, which is a natural next step toward production readiness. This was a good reminder that security decisions are not made in isolation — they have performance implications that must be considered and communicated.

The schema migration was the most technically demanding part of the enhancement. Writing a safe onUpgrade() method that adds new tables, migrates existing data from the old plain-text category column into the new normalized categories table, and then back-fills the foreign key IDs into the inventory table — all without losing a single row of user data — required careful ordering of operations and testing against an actual existing database. The instinct when dealing with a schema change is to simply drop and recreate everything, and the original onUpgrade() in the CS 360 project did exactly that. Learning to write incremental, data-preserving migrations is the kind of skill that separates a developer who has only built greenfield projects from one who understands the full lifecycle of a production application.
