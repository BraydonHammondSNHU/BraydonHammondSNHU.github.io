Narrative

Artifact Description. The artifact is the Android Inventory Manager application I originally developed in CS 360: Mobile Architecture and Programming. Written in Java for the Android platform, the application allows users to create an account, log in, and manage a personal inventory of items, each with a name, SKU, quantity, and category. Users can add new items, update quantities, delete records, and receive SMS notifications when stock falls below a defined threshold. The application was originally created during CS 360 and has since been enhanced across multiple categories for the CS 499 capstone ePortfolio.

Justification and Improvements. I selected this artifact for the algorithms and data structures category because the original application had meaningful and demonstrable inefficiencies in how it handled data retrieval, filtering, and display — and because fixing those inefficiencies gave me a genuine opportunity to apply specific data structure concepts rather than making cosmetic changes. The original application loaded all inventory items from the database with a full table scan on every screen refresh, maintained no in-memory data model, and provided no way for a user to search or sort the list. These were not just design limitations; they were algorithmic problems with measurable solutions.

The enhancement introduced four concrete algorithm and data structure improvements. First, I implemented a HashMap index that maps each item's lowercase name to its corresponding Item object. This index is rebuilt in O(n) time whenever the item list is reloaded, and it enables O(1) average-case lookup when a user searches by exact name, compared to the O(n) linear scan that would otherwise be required. Second, I implemented three sort modes — alphabetical by name, quantity ascending, and quantity descending — using Java's Collections.sort() with appropriate Comparators. The Item class was updated to implement Comparable<Item> so that the default alphabetical sort uses the natural ordering defined on the class itself, while the two quantity-based sorts use lambda Comparators. All three sorts run in O(n log n) time via Java's TimSort implementation. Third, I implemented a min-heap using Java's PriorityQueue ordered by quantity ascending to surface the lowest-stock items. All n items are inserted in O(n log n) total, and the top-k lowest-quantity items are extracted in O(k log n), making this more efficient than a full sort when only a small subset of results is needed. The results are displayed in a low-stock banner that appears automatically when items fall below the threshold. Fourth, the search field already present in the layout was wired to the ViewModel through a TextWatcher, routing each keystroke to either the O(1) HashMap exact match or an O(n) prefix scan fallback, giving users real-time filtering without repeated database queries. All complexity analysis is documented inline throughout the ViewModel code.

Course Outcomes. In Module One I planned for this enhancement to address Course Outcome 3 — designing and evaluating computing solutions that solve a given problem using algorithmic principles and computer science practices and standards appropriate to its solution, while managing the trade-offs involved in design choices — and Course Outcome 4 — demonstrating an ability to use well-founded and innovative techniques, skills, and tools in computing practices for the purpose of implementing computer solutions that deliver value and accomplish industry-specific goals.

Both outcomes were met. Course Outcome 3 is addressed most directly by the explicit trade-off analysis embedded in the implementation: the HashMap provides O(1) lookup at the cost of O(n) additional memory and the overhead of rebuilding the index on each load. The min-heap provides O(k log n) extraction of the top-k items without requiring a full sort, which is advantageous when k is small relative to n. These are not arbitrary choices — they are deliberate algorithmic decisions with documented reasoning, which is exactly what Outcome 3 requires. Course Outcome 4 is addressed by applying established computer science data structures — hash maps, priority queues, and comparator-based sorting — to a real mobile application in a way that delivers tangible user-facing value in the form of search and sort functionality.

I have no significant updates to my outcome-coverage plan. Course Outcomes 1 and 2 continue to be addressed through the code review, narratives, and documentation. Course Outcome 5 remains tied to the database enhancement.

Reflection. The most important thing I learned through this enhancement is the difference between knowing what a data structure does and knowing when and why to choose it. I was familiar with HashMaps and priority queues before starting this milestone, but working through the actual implementation forced me to think carefully about the specific access patterns of an inventory application and match the right structure to each one. A user searching by name has a fundamentally different access pattern than a user scanning for low-stock items, and the fact that those two problems call for different solutions — a hash map for the first, a heap for the second — was something I understood abstractly but had not previously applied in a concrete project.

The most challenging part of the enhancement was implementing the search correctly. My initial instinct was to run all searches through the HashMap, but I quickly realized that a HashMap can only return an exact match — if a user types “ham” looking for “Hammer,” the HashMap returns nothing. The solution was a two-tier approach: attempt the O(1) HashMap lookup first, and if it misses, fall back to an O(n) prefix scan. Documenting why this fallback exists and what its cost is was as important as implementing it, because the whole point of the complexity analysis in the narrative and code comments is to show that the trade-off was understood, not just that the feature works. Another challenge was integrating the min-heap with the LiveData architecture from the Software Engineering enhancement. The heap needs to operate on the in-memory master list rather than querying the database again, which required careful sequencing inside loadItemsForUser() to ensure the heap is always rebuilt from the most current data. Getting that ordering right — load cursor, build master list, build HashMap index, sort, post to LiveData, rebuild heap, post to low-stock LiveData — took more careful thought than I anticipated, but the result is a clean, deterministic update sequence that keeps all derived data structures in sync with the underlying source of truth.
